## 1. 들어가면서
* CPU 스케줄링은 **어떤 프로세스를 어느 시간동안 실행시킬 것인가**를 결정하는 과정이라고 생각할 수 있다.
* 스케줄링을 공부하기 이전에 두 가지 용어를 우선 정의하면 좋을 것 같다.
  **1. Nonpreemptive scheduling** : 이미 할당된 자원을 다른 프로세스가 빼앗을 수 없다는 것이다.
  **2. Preemptive scheduling** : Time slice가 소진되었거나 I/O Interrupt, system call 등이 발생하면 현 실행 프로세스로부터 강제로 CPU를 회수하여 프로세스를 스위칭하는 것이다.
* 예전에 프로세스의 다섯 가지 상태에 대해서 글을 썼을 때 참고한 그림이 있었다.

## 2. CPU Scheduling Criteria
* CPU 스케줄링이 지금까지 연구되는 이유는 시스템과 유저 관점에서의 교착 상태때문이다. 보다 자세히 설명을 하자면, 유저 입장에서는 JOB을 빨리 처리해주는 게 좋은 것이고, 시스템 입장에서는 CPU를 최대한 많이 바쁘게 돌리는 것이 중요한 것이다. 이 두가지를 동시에 만족시키는 것이 어렵기 때문에 CPU 스케줄링을 설계해야 하는 것이다.
* 그렇다면 CPU 입장과 Process 입장에서 각각 어떤 기준이 있는지 살펴보자
- **CPU Oriented(System)**  
  - Utilization: CPU를 최대한 바쁘게 하는 것
  - Throughput: 주어진 시간안에 최대한 많은 프로세스를 완료시키는 것

- **Process oriented(user)**
  - Turnaround time: 특정 프로세스를 수행하는데 걸리는 총 시간
  - Waiting time: Ready queue에서 프로세스가 실행되기를 기다리는 시간

* 결국 스케줄링 정책은 다음 두 가지 관점에서 정의된다.
1. **The selection function(which) : 어떤 프로세스를 고를까?**
2. **The Decision mode(when) : 언제 프로세스를 바꿀까?** - Nonpreemptive, Preemptive ?

## 3. CPU-Bound와 I/O-Bound
* 프로세스는 명령어(Instruction)을 수행하다가 I/O를 기다렸다가, 해결이 되면 다시 CPU로 왔다가를 반복한다.   
<img src = "https://user-images.githubusercontent.com/61929745/113540680-9e5b5080-961b-11eb-8191-d74c41847448.png" width = 400 height = 400 >   
출처: https://jhnyang.tistory.com/25

* 각 사이클은 CPU를 사용하는 burst time과 그 뒤에 I/O burst time이 존재하고, 어떤 프로세스의 CPU burst time이 10초라고 하면, 이 프로세스의 어떤 작업이 완료되기 위해서는 CPU가 10초동안 이 프로세스를 작업해줘야 한다는 뜻이다. 그 다음에 I/O burst가 왔다는 것은 10초동안 이 작업을 하고, I/O Interface로부터 작업이 필요한 시간이 존재한다는 것이다.
* **이 때, CPU-Bound process는 말그대로 CPU에서 처리해야 하는 일이 많은 프로세스를 의미하고 I/O-Bound process는 I/O 작업에서 필요한 시간이 많은 프로세스를 의미한다.**

---
## 4. Priority
* Priority = 우선순위란 말그대로 우선순위다. 만약 스케줄링을 우선순위의 관점에서 하게 된다면 우선순위가 높은 프로세스가 먼저 할당이 되는 식이다.
* 하지만 우선순위만을 기반으로 하게 되면 **Starvation problem**이 생길 수 있는데, 말그대로 우선순위가 낮은 프로세스는 배고파 죽을 때까지 할당을 받지 못한다는 것이다.
